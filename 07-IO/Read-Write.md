# 图解 Rust I/O：Read 与 Write

在 Rust 中，`Read` 和 `Write` 是处理 I/O 的核心 trait。它们将各种数据源（如文件、网络、内存）统一抽象为“字节流”，屏蔽了底层的实现细节，让你能用一套 API 在不同介质间移动数据。

---

## 1. 内存布局：BufReader 与 BufWriter

普通读写每次都会触发系统调用（System Call），在大批量读写小数据时效率极低。`BufReader` 和 `BufWriter` 通过在内存中增加一个 **Buffer (缓冲区)** 来解决这个问题。

它们在 **Stack** 上持有指针，而实际的缓冲区则分配在 **Heap** 上。这样，连续的小次读写会被暂存在内存中，只有缓冲区填满或读空时，才会真正触发系统调用与硬件交互。

## 2. 逻辑流转：std::io::copy 的搬运机制

相比手动编写 `loop` 读写，`std::io::copy` 是标准库中最高效的数据搬运工具。它会自动在 **Stack** 上分配一个 **8KB 的中转数组**，通过循环执行“读取 ➜ 填充 ➜ 写出”直到 EOF，极大地减少了用户态与内核态的切换频率。此外，在 Linux 等支持 `sendfile` 的平台上，它甚至能实现内核空间直接移动的“零拷贝”优化。

## 3. 核心行为：缓冲与刷新 (Flush)

写入操作通常比读取更“危险”，因为数据可能还留在内存里。`BufWriter` 就像一个蓄水池，只有攒够了水才会开闸。

![BufWriter 缓冲与刷新](./imgs/buf_writer_flush.svg)

### 写入过程拆解

1.  **Buffer Write**：当你调用 `writer.write()` 或 `writeln!()` 时，数据只是通过 `memcpy` 存入了用户空间的缓冲区，并没有真正落盘。这一步非常快。
2.  **Flush**：当缓冲区溢满、手动调用 `.flush()`、或者 `BufWriter` 被销毁（Drop）时，数据才会被发送给底层设备。

### 生产环境 vs. 教学示例

在实际工程中，`BufWriter` 的用法与简单示例有显著差异：

- **合并小写入**：生产代码通常在循环中使用 `writeln!` 或 `write!` 写入大量小数据（如日志行）。`BufWriter` 会将这些成千上万次的小写入合并为极少数的系统调用。
- **显式 Flush 的必要性**：虽然 `BufWriter` 在销毁（Drop）时会自动尝试刷新，但 `Drop` 过程中的错误会被静默忽略。在处理关键业务数据时，**必须显式调用 `.flush()?`** 以捕获磁盘满或权限不足等 IO 错误。
- **容量调优**：默认的 8KB 缓冲区适合大多数场景。但在高性能存储（如 NVMe SSD）或高吞吐任务中，开发者常使用 `BufWriter::with_capacity(128 * 1024, file)` 将缓冲区扩大到 128KB 或更多，以进一步减少上下文切换。

## 4. 设计哲学

Rust 的 I/O 系统设计遵循了几个核心原则：

*   **零成本抽象**：`Read`/`Write` 默认使用静态分发，没有任何虚函数开销。只有当你需要灵活性时（如 `dyn Read`），才支付动态分发的代价。
*   **显式控制**：缓冲功能不是默认内置在文件对象中的，而是通过包装器（Wrapper）显式提供。这符合 Rust “不隐瞒开销”的原则。
*   **安全性**：所有权和借用规则保证了在异步或多线程 I/O 场景下，缓冲区不会被非法访问，从根本上杜绝了数据竞争。
